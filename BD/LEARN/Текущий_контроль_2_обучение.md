## 1. Триггер — что это

**Триггер** — правило: при событии с таблицей (`INSERT` / `UPDATE` / `DELETE`) СУБД **сама** вызывает вашу функцию.

- Событие задаётся в `CREATE TRIGGER`, например `BEFORE INSERT ON products`.
- Код пишут в функции с **`RETURNS TRIGGER`** (триггерная функция в PostgreSQL).
- Связка: `CREATE TRIGGER ... EXECUTE PROCEDURE имя_функции();` (в версиях 14+ допускают и `EXECUTE FUNCTION` — смысл тот же).

Строка меняется → срабатывает триггер → выполняется ваш код.

---

## 2. Слова, без которых не разобрать контрольную

| Что | Простыми словами |
|-----|------------------|
| `BEFORE` | Функция вызывается **до** фиксации строки. Можно изменить `NEW` или **отменить** команду через `RAISE EXCEPTION`. |
| `AFTER` | Вызывается **после** успешного изменения строки. Удобно писать в журнал, когда «новое» уже стало фактом в таблице. |
| `NEW` | **Новая** версия строки (`INSERT`, `UPDATE`). При `DELETE` нет. |
| `OLD` | **Старая** версия (`UPDATE`, `DELETE`). При чистом `INSERT` нет. |
| `RETURN NEW` / `RETURN OLD` | На `BEFORE INSERT/UPDATE` обычно возвращают `NEW` (возможно изменённый). На `BEFORE DELETE` — `OLD`. На `AFTER` в PostgreSQL часто пишут `RETURN NEW` или `RETURN COALESCE(NEW, OLD)` — так принято для типа результата. |
| `RAISE EXCEPTION` | **Ошибка**: текущая команда откатывается, клиент видит текст — то, что просят в заданиях 1, 3 и при «ручном» `total` в задании 4. |
| `UPDATE OF имя_столбца` | Триггер срабатывает только если в `UPDATE` реально затронули перечисленные колонки — в задании 2 это экономит лишние вызовы при правке ФИО. |

## 3. Сводка по заданиям контрольной

| № | Таблица(ы) | Момент | Задача |
|---|------------|--------|--------|
| 1 | `products` | `BEFORE INSERT OR UPDATE` | Не пустить `price <= 0`, сообщение строго из условия. |
| 2 | `employees` → `salary_log` | `AFTER UPDATE OF salary` | Писать историю смены зарплаты только если значение **реально** изменилось. |
| 3 | `customers`, `orders` | `BEFORE DELETE` на клиентах | Запретить удаление, если есть хотя бы один заказ. |
| 4 | `order_items` | `BEFORE INSERT OR UPDATE` | Всегда `total = quantity * price`; не давать «втихаря» записать другой `total`. |

## 4. Задание 1 — цена товара только положительная

**Идея:** пока строка **ещё не записана**, проверяем `NEW.price`. Если нарушение — `RAISE EXCEPTION` с текстом *«Цена товара должна быть положительной»*.

**Почему `BEFORE`:** отсечь неверные данные **до** появления строки в таблице; не хранить «нулевые» и отрицательные цены.

**Почему `INSERT OR UPDATE`:** и при добавлении, и при смене цены правило одно и то же.

---

## 5. Задание 2 — журнал изменений зарплаты

**Идея:** когда в `employees` **уже** обновили строку и поле `salary` действительно другое, добавить строку в `salary_log`: кто, было, стало, `NOW()`.

**Почему `AFTER`:** логируем **факт** после успешного обновления; для записи в другую таблицу это естественный момент.

**Почему `IS DISTINCT FROM`:** корректно отличает «было NULL — стало число» и наоборот; обычное `<>` с `NULL` даёт `UNKNOWN` и не сработает в условии так, как ожидают.

**Почему не логировать «пустой» апдейт:** `UPDATE ... SET salary = salary` не должен плодить строки в журнале — в условии прямо сказано «только если зарплата действительно изменилась».

---

## 6. Задание 3 — нельзя удалить клиента с заказами

**Идея:** перед удалением строки клиента смотрим `orders`: есть ли `customer_id = OLD.id`. Если да — `RAISE EXCEPTION` с текстом из задания.

**Почему `BEFORE DELETE`:** операцию можно **заблокировать** до физического удаления.

**Почему `RETURN OLD`:** на удалении возвращают старую версию строки; для PostgreSQL это нормальная форма для `BEFORE DELETE`.

**Связь таблиц:** в условии `orders.customer_id` ссылается на `customers(id)` — целостность данных на уровне схемы; триггер добавляет **бизнес-правило** «не терять клиента, пока живы заказы».

---

## 7. Задание 4 — `total` только из формулы

**Идея:** перед вставкой/обновлением считаем `quantity * price` и записываем в `NEW.total`. Если пользователь передал `total`, который **не совпадает** с формулой — считаем попытку «ручного» ввода и рвём команду исключением (вариант в [`контрольная работа_2.md`](../контрольная%20работа_2.md)).

**Альтернатива на защите:** некоторые преподаватели просят **молча перезаписывать** `total` без проверки — тогда достаточно одной строки `NEW.total := NEW.quantity * NEW.price`; уточняйте формулировку.

**Почему `BEFORE`:** нужно успеть подменить/вычислить поле **до** записи.

---

## 8. Что могут спросить на защите

- **Чем `BEFORE` отличается от `AFTER`?** До записи — правим `NEW` или отменяем операцию; после — фиксируем факт и пишем в другие таблицы.
- **Почему для проверки цены `BEFORE`, а для лога зарплаты `AFTER`?** Цену не даём попасть в таблицу; лог — про уже случившееся изменение.
- **Зачем `UPDATE OF salary`?** Триггер не дергается понапрасну при обновлении других колонок.
- **Почему в логе сравнивают через `IS DISTINCT FROM`?** Корректная работа с `NULL`.
- **В `DELETE` что такое `OLD.id`?** Это идентификатор удаляемой строки — по нему ищем «висящие» заказы.
- **Зачем `DROP TRIGGER` перед повторным созданием?** Иначе при втором запуске скрипта будет конфликт имён.

---

## 9. Перед защитой — три фразы

1. **Триггер** — автоматический вызов функции с `RETURNS TRIGGER` при изменении таблицы; внутри доступны `NEW`, `OLD` и тип операции.
2. **Контрольная 2** — четыре типовых паттерна: запрет по условию до записи, аудит после обновления конкретного поля, запрет удаления по связям, вычисляемое поле до записи.
3. **Текст исключений** в заданиях 1 и 3 должен **совпасть** с формулировкой в листе задания — на это смотрят при проверке.

Если нужно, можно расписать **построчно** только одну функцию — напишите номер задания.
