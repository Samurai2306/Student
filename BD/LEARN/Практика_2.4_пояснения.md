# Практика №2.4 — подробные пояснения к заданиям (дата/время)

Эта практика про хранение событий «вход/выход» и расчёты на основе времени: отработанные часы, опоздания, зарплата.

## Задание 1 — Таблица `рабочий_день` + тестовые данные

- **Что делаем**: создаём таблицу событий прохода через систему доступа:
  - `employee_id` — сотрудник
  - `event_ts` — дата/время события
  - `reader_value` — тип события: `1` (вход), `2` (выход)
  Дальше заполняем тестовыми событиями для нескольких сотрудников за 1–2 дня.
- **Зачем**:
  - «событийная модель» (event log) — стандартная схема хранения проходов/логов;
  - на такой структуре удобно строить любые расчёты: часы работы, нарушения режима, аналитика.
- **Почему так**:
  - **Одна таблица событий** вместо «колонки вход/выход»:
    - события могут быть в разном количестве (ошибка считывателя, повторный проход);
    - можно хранить точное время каждого действия.
  - **`CHECK (reader_value IN (1,2))`**: защита от мусорных значений.
  - **Составной `PRIMARY KEY`** (employee_id, event_ts, reader_value): не даёт случайно вставить идентичное событие второй раз.

На что обратить внимание:
- если в реальных данных будет «вход-вход» без выхода или «выход» без входа — расчёты надо усложнять. В учебном варианте предполагается корректная последовательность.

## Задание 2 — `week_work_hours(employee_id)` (отработанные часы за неделю + статус)

- **Что делаем**:
  1) берём события сотрудника по времени,
  2) складываем все интервалы «вход(1) → следующий выход(2)»,
  3) переводим секунды в часы,
  4) сравниваем с 40 часами и выдаём статус: «Меньше нормы / Норма / Больше нормы».
- **Зачем**:
  - типовой расчёт табеля/учёта рабочего времени;
  - тренировка оконных функций (`LEAD`) и работы с интервалами.
- **Почему так**:
  - **`LEAD(event_ts)`**: позволяет для каждой строки события увидеть следующую временную метку без процедурного цикла.
  - **Пара «1 → 2»**: считаем только корректные пары вход-выход, остальное игнорируем.
  - **`EXTRACT(EPOCH FROM (next_ts - event_ts))`**: получаем длительность в секундах и суммируем.
  - **`COALESCE(hours,0)`**: если событий нет — 0 часов.

Типичные ошибки:
- считать просто `MAX(out) - MIN(in)` за день — это неверно, если есть обед или несколько выходов/входов.

## Задание 3 — `calc_salary_with_tardiness(employee_id, base_salary)` (зарплата с учётом опозданий)

- **Что делаем**:
  1) для каждого дня сотрудника определяем ключевые точки:
     - `first_in` — первый вход,
     - `lunch_out` — уход на обед,
     - `lunch_in` — возвращение с обеда,
     - `last_out` — последний выход,
  2) считаем «минуты опоздания» по правилам:
     - вход позже 09:00,
     - уход на обед раньше 13:00,
     - возвращение с обеда позже 14:00,
     - уход раньше 18:00,
  3) суммируем опоздания за неделю,
  4) считаем коэффициент \(A\): старт 1, за каждые 10 минут −0.05, но не ниже 0,
  5) зарплата = оклад + оклад * A.
- **Зачем**:
  - пример «бизнес-правил» на времени: штрафы/бонусы;
  - тренировка работы с `DATE`, `TIME`, интервалами и математикой.
- **Почему так**:
  - **`days`**: сначала получаем список дат, чтобы считать по дням (иначе всё смешается).
  - **`day_marks`**: отдельный слой, где из событий выбираются нужные метки (MIN/MAX + фильтры).
  - **`GREATEST(0, ...)`**: штраф считается только когда нарушение есть (не допускаем отрицательных минут).
  - **`FLOOR(total_late_minutes/10.0)`**: правило «за каждые 10 минут» — это целые десятки минут.
  - **`GREATEST(0, 1 - 0.05 * tens)`**: ограничение \(A \ge 0\).

На что обратить внимание:
- в отчёте заложены «разумные дефолты» через `COALESCE` (если метки нет, считаем её равной нормативному времени). Это делает функцию устойчивее на неполных данных, но в реальной системе лучше явно фиксировать ошибки данных.
