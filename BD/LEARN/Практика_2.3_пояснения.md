# Практика №2.3 — подробные пояснения к заданиям (PL/pgSQL)

Эта практика про процедурную логику в PostgreSQL (PL/pgSQL): условия, циклы, обновления, проверки аномалий и запись результатов.

## Задание 1 — `apply_property_discounts()` (снижение цены по давности и рейтингу)

- **Что делаем**: уменьшаем `properties.price_rub` для объектов, которые давно размещены и имеют низкую среднюю оценку:
  - старше 6 месяцев и avg_rating < 6 → −5%
  - старше 9 месяцев и avg_rating < 5 → −10%
  - старше 12 месяцев и avg_rating < 4 → −20%
  Функция возвращает количество изменённых строк.
- **Зачем**:
  - пример «массового обновления» данных по бизнес-правилам;
  - тренировка агрегатов (средний рейтинг) + условий скидки.
- **Почему так**:
  - **`LEFT JOIN property_ratings`**: чтобы можно было посчитать `AVG` по оценкам.
  - **`AVG(...)` + `GROUP BY`**: средняя оценка считается по всем оценкам объекта.
  - **CTE `rated`**: отделяет шаг «посчитать рейтинг» от шага «решить скидку».
  - **CTE `to_update`**: содержит только `id` и размер скидки (удобно читаться).
  - **`UPDATE ... FROM to_update`**: массовое обновление без цикла — быстрее и правильнее, чем обновлять по одной строке.
  - **`GET DIAGNOSTICS ... ROW_COUNT`**: возвращаем именно число изменённых строк, как требует задание.

На что обратить внимание:
- если запускать функцию несколько раз подряд, цена будет уменьшаться снова и снова. В реальной системе обычно ставят флаг/дату последнего пересчёта, чтобы не применять скидку повторно.

## Задание 2 — `find_sales_date_anomalies()` (продажа раньше публикации)

- **Что делаем**: ищем объекты, у которых `sales.sale_date < properties.created_at`, и выводим их адреса. Если таких нет — возвращаем строку «Аномалий не найдено».
- **Зачем**:
  - пример контроля качества данных;
  - учит сравнивать даты и возвращать набор строк из функции.
- **Почему так**:
  - **Проверка «аномалия»** — это типичный data quality rule.
  - **`RETURNS TABLE(result TEXT)`**: функция может вернуть много адресов (набор строк).
  - **Цикл `FOR rec IN SELECT ... LOOP`**: удобен, когда нужно пройти по найденным строкам и вернуть их одну за другой через `RETURN NEXT`.
  - **Флаг `any_found`**: нужен, чтобы понять, были ли строки; если нет — вернуть сообщение.

## Задание 3 — `check_property_area_consistency()` (общая площадь vs сумма комнат)

- **Что делаем**: сравниваем `properties.total_area_m2` с суммой `property_rooms.room_area_m2` по каждому объекту.
  - если совпадает для всех — возвращаем строку «ВСЕ ПЛОЩАДИ СООТВЕТСТВУЮТ»;
  - если есть расхождения — возвращаем адрес и величину расхождения.
- **Зачем**:
  - пример проверки согласованности данных между таблицами (родитель-дети);
  - тренировка агрегатов + `HAVING`.
- **Почему так**:
  - **`LEFT JOIN property_rooms`**: если комнат нет, сумма комнат считается как 0.
  - **`COALESCE(SUM(...), 0)`**: чтобы не получить `NULL` при отсутствии комнат.
  - **`HAVING ABS(diff) > 0.01`**: сравнение чисел с плавающей точкой/десятичной частью лучше делать с допуском (из-за округлений). 0.01 — «1 сантиметр» в квадратных метрах условно; можно менять.
  - **Возврат набора строк**: потенциально несоответствий может быть много.

## Задание 4 — `recalc_sales_commission()` (комиссия риэлтору по цене продажи)

- **Что делаем**: для каждой продажи пересчитываем комиссию и записываем в `sales.commission_rub`:
  - до 1 млн → 2%
  - 1–3 млн → 1.9%
  - от 3 млн → 1.7%
  Функция возвращает количество обновлённых строк.
- **Зачем**:
  - пример массового пересчёта производного значения (commission) по тарифной сетке;
  - закрепление `CASE WHEN` внутри `UPDATE`.
- **Почему так**:
  - **`UPDATE sales SET commission = price * CASE ... END`**: самый прямой способ пересчитать всем строкам.
  - **`ROUND(..., 2)`**: деньги обычно округляют до копеек.
  - **`GET DIAGNOSTICS`**: число обработанных (обновлённых) записей соответствует требованию.
