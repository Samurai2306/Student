# Практика №2.1 — подробные пояснения к заданиям

Ниже — пояснения **что делает каждое задание**, **зачем оно нужно** и **почему код написан именно так** (на уровне логики и механизмов PostgreSQL).

## Задание 1 — `calculate_age(birth_date)` (возраст по дате рождения)

- **Что делаем**: создаём функцию, которая по `DATE` (дата рождения) возвращает возраст в годах на текущую дату.
- **Зачем**: возраст часто нужен для отчётов/валидаций (например, «18+»), и удобно иметь это как функцию, чтобы использовать в `SELECT`, `WHERE`, `CHECK`, представлениях.
- **Почему так**:
  - **Проверка `NULL`**: если дата рождения неизвестна, корректнее вернуть `NULL`, а не «0», потому что «0» — это реальное число и вводит в заблуждение.
  - **`AGE(CURRENT_DATE, birth_date)`**: `AGE` возвращает `interval`, учитывая месяцы и дни (то есть корректно обрабатывает, был ли уже день рождения в этом году).
  - **`EXTRACT(YEAR FROM ...)`**: из интервала берём количество лет.
  - **Почему не `CURRENT_DATE - birth_date`**: вычитание дат даёт количество дней, и перевод в годы будет грубым и часто неверным.

## Задание 2 — `categorize_order(order_amount)` (категория по сумме)

- **Что делаем**: функция принимает сумму заказа и возвращает текстовую категорию: «МАЛЫЙ/СРЕДНИЙ/КРУПНЫЙ/ОСОБО КРУПНЫЙ».
- **Зачем**:
  - быстро классифицировать заказы для отчётности и BI (сколько «крупных» заказов);
  - использовать в аналитике: `GROUP BY categorize_order(amount)`.
- **Почему так**:
  - **Проверка `NULL`**: неизвестная сумма → неизвестная категория → `NULL`.
  - **Проверка на отрицательное**: отрицательная сумма — это аномалия (ошибка ввода/данных). Поэтому лучше **сразу остановить** выполнение через `RAISE EXCEPTION`, чем молча вернуть категорию.
  - **Границы интервалов**: построены каскадом `IF/ELSIF`, чтобы условия были взаимоисключающие и читались сверху вниз (малые → большие).
  - **Почему PL/pgSQL, а не SQL**: из-за `IF` и исключения проще и понятнее использовать `plpgsql`.

## Задание 3 — `generate_order_numbers(count)` (генерация набора номеров)

- **Что делаем**: функция генерирует `count` строк с номерами формата `ORD-YYYYMMxxxxxx`, где `xxxxxx` — 6 случайных цифр.
- **Зачем**:
  - генерировать тестовые данные (нагрузочные тесты, демо);
  - быстро получать набор строк без отдельной таблицы.
- **Почему так**:
  - **`RETURNS TABLE(...)`**: функция возвращает не одно значение, а набор строк (табличный результат).
  - **`FOR i IN 1..count LOOP`**: самый прямой способ сделать цикл генерации.
  - **`RANDOM()` + `FLOOR()`**: получаем целое 0..999999.
  - **`LPAD(..., 6, '0')`**: чтобы всегда было ровно 6 цифр (например, `000042`).
  - **`RETURN NEXT`**: добавляет одну строку в результат табличной функции.
  - **Проверка `count <= 0`**: если запросили 0 или отрицательно — корректно вернуть пустой набор.

## Задание 4 — `transfer_money(from_account_id, to_account_id, amount)` (перевод средств)

- **Что делаем**: функция переводит деньги между счетами:
  1) проверяет входные данные,
  2) проверяет, что счета существуют,
  3) проверяет достаточность средств,
  4) обновляет балансы,
  5) пишет запись в `transactions`,
  6) возвращает `TRUE/FALSE`.
- **Зачем**:
  - это классический пример «прикладной бизнес-операции» в БД;
  - учит работать с конкурентностью, блокировками и обработкой ошибок.
- **Почему так**:
  - **Почему нужны проверки**: без них легко получить «перевод самому себе», отрицательную сумму, перевод с несуществующего счёта.
  - **`SELECT ... FOR UPDATE`**: блокирует строки счетов, чтобы два параллельных перевода не «перетёрли» баланс (гонка).
  - **Почему нельзя просто `UPDATE` без блокировок**: при конкурентных операциях возможны некорректные промежуточные состояния.
  - **Запись в `transactions`**: фиксируем факт операции (аудит/история).
  - **`EXCEPTION WHEN OTHERS THEN RETURN FALSE`**: если что-то пошло не так (ошибка, нарушение ограничений), функция возвращает `FALSE`. Это упрощает внешний код (можно проверять «успех/неудача»).
