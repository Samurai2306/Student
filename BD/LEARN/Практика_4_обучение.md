## 1) Что такое транзакция

**Транзакция** — группа SQL-команд, которая выполняется как единое целое:

- **ACID (очень кратко)**:
  - **Atomicity**: либо все изменения применились, либо ни одно.
  - **Consistency**: переход из одного корректного состояния базы в другое (в рамках ограничений/правил).
  - **Isolation**: параллельные транзакции не «ломают» друг друга; степень изоляции задаётся уровнем.
  - **Durability**: после `COMMIT` данные сохраняются надёжно.

В PostgreSQL явная транзакция начинается `BEGIN` и заканчивается `COMMIT` (зафиксировать) или `ROLLBACK` (откатить).

## 2) MVCC «на пальцах»: почему у строки есть версии

PostgreSQL использует **MVCC (Multi-Version Concurrency Control)** — многоверсионность данных.

Идея: вместо того чтобы «переписывать строку на месте», база часто создаёт **новую версию**, а старую помечает как больше неактуальную для будущих снимков.

### `xmin` и `xmax`

В каждой физической версии строки (tuple) есть служебные поля:

- **`xmin`**: id транзакции, которая **создала** эту версию строки.
- **`xmax`**: id транзакции, которая **удалила** эту версию (или «закрыла» её обновлением, т.е. сделала старой версией).

Запрос из отчёта:

```sql
SELECT xmin, xmax, * FROM a;
```

### Почему `UPDATE` меняет и `xmin`, и `xmax`

`UPDATE` в MVCC обычно означает:

1) старая версия получает `xmax = xid_обновляющей_транзакции` (она «закрыта»);
2) появляется новая версия с новым `xmin`.

Поэтому во второй сессии часто видно:

- значение данных ещё старое (новая версия не закоммичена),
- но `xmax` у старой версии уже изменился (факт обновления зафиксирован как намерение текущей транзакции).

## 3) Снимок данных (snapshot) и видимость строк

Каждая транзакция читает данные не «как сейчас в мире», а через **снимок** (snapshot): набор правил «какие транзакции уже считаются завершёнными» и «какие изменения можно видеть».

Отсюда типичный эффект практики:

- **Терминал A** внутри своей транзакции видит собственные изменения.
- **Терминал B** до `COMMIT` в A их не видит.

## 4) Уровни изоляции, которые вы реально демонстрируете

Проверка уровня:

```sql
SHOW transaction_isolation;
```

### `READ COMMITTED` (обычно по умолчанию)

- Снимок берётся **на каждый оператор**.
- Поэтому в одной и той же транзакции B два одинаковых `SELECT` могут вернуть разный результат, если между ними кто-то сделал `COMMIT`.

Это и есть демонстрация **неповторяющегося чтения** (non-repeatable read) в рамках практики.

### `REPEATABLE READ`

- Снимок берётся **один раз на всю транзакцию**.
- Поэтому повторный `SELECT` в той же транзакции B не увидит строку, которую A добавил и закоммитил **после старта** транзакции B.

В отчёте вы наблюдаете именно это: «во втором окне изменения не видны до завершения транзакции».

## 5) CLOG и `pg_xact_status(xid)`

PostgreSQL хранит информацию о завершённых транзакциях в системных структурах (традиционно это называют **CLOG/commit log**).

### Как получить `xid` «прямо сейчас»

```sql
SELECT txid_current();
```

### Как спросить статус по номеру транзакции

```sql
SELECT pg_xact_status('<xid>');
```

Типовая логика практики:

- после `BEGIN` и до `COMMIT` — транзакция «в процессе»;
- после `COMMIT` — «зафиксирована»;
- после `ROLLBACK` — «откачена/aborted».

Важно: в методичке может быть пример с конкретным числом `1598`, но у вас оно будет **другим** — поэтому в отчёте удобно получать `xid` через `txid_current()`.

## 6) Блокировки: зачем они нужны, если есть MVCC

MVCC решает многие конфликты чтения/записи, но **не отменяет блокировки**:

- Изменение структуры таблицы (DDL), создание индекса, некоторые виды сканирования и обновления требуют режимов блокировок.
- Блокировки предотвращают ситуации, когда одна операция делает невозможной корректность другой.

### Где смотреть блокировки

```sql
SELECT locktype,
       transactionid,
       mode,
       relation::regclass AS obj
FROM pg_locks
WHERE pid = <pid_процесса>;
```

`pid` берётся так:

```sql
SELECT pg_backend_pid();
```

### Почему `UPDATE` и `CREATE INDEX` конфликтуют в практике

Упрощённо:

- `UPDATE` ставит блокировку вроде **`RowExclusiveLock`** на таблицу (изменение данных).
- `CREATE INDEX` требует более «сильной» совместимости и может запросить режимы (в методичке упоминается `ShareLock`), которые **не совместимы** с удерживаемыми блокировками другой транзакции.

Отсюда наблюдение: `CREATE INDEX` во втором терминале «подвисает», пока первый не сделает `COMMIT`.

## 7) Что обычно спрашивают на защите (готовые ответы)

- **Почему во второй сессии не видно обновления до `COMMIT`?**  
  Потому что другая транзакция работает с собственным снимком; незакоммиченные версии строк невидимы для неё.

- **Почему при `READ COMMITTED` повторный `SELECT` может показать новые строки?**  
  Потому что на этом уровне снимок обновляется на каждый оператор: второй `SELECT` видит то, что было закоммичено между запросами.

- **Чем `REPEATABLE READ` отличается по ощущению?**  
  Вся транзакция читает один и тот же снимок, поэтому повторные чтения «стабильны» и не видят изменения, закоммиченные после старта транзакции.

- **Что такое `xmin/xmax`?**  
  Служебные поля версии строки: `xmin` — кто создал, `xmax` — кто удалил/закрыл обновлением.

- **Почему `UPDATE` — это «две строки»?**  
  В MVCC создаётся новая версия, а старая помечается как устаревшая (через `xmax`), чтобы параллельные транзакции могли читать согласованно.

- **Почему в DDL возникает ожидание?**  
  Потому что блокировки разных режимов несовместимы; `pg_locks` позволяет увидеть, кто кого держит и кто кого ждёт.
