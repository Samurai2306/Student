# Практики 3.1–3.2 — обучение к отчёту по триггерам

Разбор к отчёту [`Практика_3.1-3.2_отчёт.md`](../Практика_3.1-3.2_отчёт.md): простыми словами, что такое триггеры, что делает каждое задание и что могут спросить на защите.

---

## 1. Триггер — что это вообще

**Триггер** — это правило вида: «когда с таблицей происходит событие (вставка, обновление, удаление), база **сама** вызывает твою функцию».

- Событие задаётся в `CREATE TRIGGER`: например `AFTER INSERT ON sales` — после добавления строки в «Продажи».
- Работу пишут в **функции с типом результата `RETURNS TRIGGER`** (в PostgreSQL это как раз «триггерная функция»).
- Триггер **подключают** к этой функции: `EXECUTE PROCEDURE имя_функции()`.

Итого: таблица меняется → срабатывает триггер → выполняется твой код.

---

## 2. Слова, без которых не разобрать код

| Что | Простыми словами |
|-----|------------------|
| `BEFORE` | Функция вызывается **до** того, как строка реально запишется. Можно изменить `NEW` (например подправить телефон) или отменить операцию через `RAISE EXCEPTION`. |
| `AFTER` | Вызывается **после** успешной записи. Удобно писать в журнал, крутить бонусы — когда сама продажа уже с id в базе. |
| `NEW` | **Новая** версия строки (при `INSERT` и `UPDATE`). В `DELETE` для текущей строки `NEW` нет. |
| `OLD` | **Старая** версия (при `UPDATE` и `DELETE`). При чистом `INSERT` `OLD` нет. |
| `TG_OP` | Строка: `'INSERT'`, `'UPDATE'` или `'DELETE'` — какая операция сейчас выполняется. |
| `RETURN NEW` / `RETURN OLD` | Для `BEFORE` обычно возвращают `NEW` (или изменённый `NEW`). Для `AFTER` часто пишут `RETURN COALESCE(NEW, OLD)` — так принято, чтобы тип совпадал. |
| `RAISE WARNING` | **Предупреждение**: операция **не отменяется**, клиент видит сообщение (в зависимости от настроек клиента). |
| `RAISE NOTICE` | Похоже на служебное **уведомление** (часто в консоль `psql`). Операция тоже не отменяется. |
| `RAISE EXCEPTION` | **Ошибка**: текущая команда **откатывается** (вставка/обновление не проходит), пользователь видит текст ошибки. |

## 3. Блок «Схема и подготовка» в отчёте

**Зачем `CREATE TABLE IF NOT EXISTS`**

Чтобы скрипт можно было запускать на пустой базе: таблицы создадутся, если их ещё не было.

**Зачем два блока `DO $$ ... ALTER TABLE ... EXCEPTION WHEN duplicate_column`**

В PostgreSQL долго не было «ADD COLUMN IF NOT EXISTS». Если колонка `phone` уже есть, голый `ALTER` упадёт. Блок ловит исключение «колонка уже есть» и просто ничего не делает — удобно для повторного запуска.

**`realtor_bonuses`**

Одна строка на риэлтора: сколько бонусов накоплено. Ключ — `realtor_id` (связь с `realtors`).

**`sales_journal`**

Аудит: когда, какая операция (`INSERT`/`UPDATE`/`DELETE`), кто (`CURRENT_USER`), с каким `sale_id`.

**`DROP TRIGGER` / `DROP FUNCTION`**

Чтобы **переустановить** триггеры: сначала снимаем старые имена, потом создаём заново. Иначе при втором запуске отчёта будет ошибка «уже существует».

## 4. Практика 3.1 — по заданиям

### Задание 1 — разница цен больше 20%

**Идея:** в карточке объекта заявлена цена `properties.price_rub`, а в продаже — `sales.sale_price_rub`. Считаем **долю отклонения от заявленной**: `|продажа − заявленная| / заявленная`. Если больше `0.20`, выводим предупреждение.

**Почему `BEFORE INSERT OR UPDATE`:** проверка при каждой попытке записать или изменить продажу.

**Почему не делим на ноль:** если заявленная цена `NULL` или `0`, сравнение бессмысленно — выходим через `RETURN NEW` без предупреждения.

**`%%` в тексте `RAISE WARNING`:** в формате строки `%` — это placeholder; чтобы вывести знак процента, в языке пишут `%%`.

---

### Задание 2 — нельзя продать один объект дважды

**Идея:** в `sales` уже есть строка с таким `property_id` — новую (или смену `property_id` на уже проданный) **запрещаем**.

**Почему `RAISE EXCEPTION`:** по условию задачи нужно и сообщение, и **запрет** операции.

**`s.id IS DISTINCT FROM NEW.id`:** при **обновлении** той же строки продажи её же id не считаем «чужой продажей». При **вставке** у `NEW.id` ещё может не быть значения до срабатывания триггеров — в PostgreSQL для `SERIAL` на `BEFORE INSERT` id иногда уже есть; `IS DISTINCT FROM` аккуратно обрабатывает `NULL`.

---

### Задание 3 — комнаты «перевешивают» общую площадь

**Идея:** перед вставкой новой комнаты считаем: сумма площадей **уже существующих** комнат этого объекта **плюс** площадь **вставляемой** строки. Сравниваем с `properties.total_area_m2`. Если сумма **больше** общей площади (с маленьким допуском `0.01` из‑за дробей) — `RAISE WARNING` с числом **на сколько м² превышение**.

**Почему только `BEFORE INSERT`:** в методичке проверка при **добавлении** записи в структуру комнат.

**Вставка всё равно проходит:** используется `WARNING`, не `EXCEPTION` — данные сохраняются, но пользователь видит сигнал.

---

### Задание 4 — продажа раньше объявления

**Идея:** дата размещения — `properties.created_at`, дата продажи — `sales.sale_date`. Если день продажи **раньше** дня размещения — предупреждение.

**Сравнение через `::DATE`:** чтобы сравнивать календарные дни, а не время суток внутри одного дня (если нужно именно «день к дню»).

---

### Задание 5 — журнал по продажам

**Идея:** на каждый `INSERT`/`UPDATE`/`DELETE` по `sales` — одна строка в `sales_journal`.

**Почему `AFTER`:** для `INSERT` в логе нужен уже присвоенный `NEW.id` — он гарантированно есть после вставки.

**`CURRENT_USER`:** имя пользователя базы данных (роль), под которой выполняется команда.

## 5. Практика 3.2 — по заданиям

### Задание 1 — бонусы 5% и откат при удалении продажи

**Идея:** с продажи риэлтору начисляется `5%` от `sale_price_rub`. При **новой** продаже риэлтора строка в `realtor_bonuses` создаётся; при **повторных** — сумма **увеличивается** (`ON CONFLICT DO UPDATE`).

**При удалении продажи:** из бонуса вычитается те же 5% от удалённой цены; `GREATEST(..., 0)` не даёт уйти в минус из‑за округлений.

**Почему два триггера:** один на `AFTER INSERT`, другой на `AFTER DELETE` — разные события, разный код (`NEW` vs `OLD`).

---

### Задание 2 — «информировать» о превышении потолка бонуса

**Идея:** после изменения `bonus_amount` смотрим: если больше заданного **максимума** (в отчёте константа 500 000) — `RAISE NOTICE`.

**NOTICE vs WARNING:** оба не отменяют транзакцию; `NOTICE` чаще для «информации», `WARNING` — для более серьёзного сигнала. Для формулировки «информировать» в отчёте взят `NOTICE`; на защите можно сказать, что при желании заменяется на `WARNING`.

---

### Задание 3 — паспорт по маске

**Идея:** не пустое значение должно совпасть с шаблоном **ровно** четыре цифры, пробел, шесть цифр: `1234 567890`.

**Регулярка `^\d{4} \d{6}$`:** `^`/`$` — начало и конец строки; `\d` — цифра; пробел между группами **один**.

**Пустое и `NULL`:** разрешаем (не мешаем вставке); если преподаватель захочет «обязательное поле» — условие меняют.

**`RAISE EXCEPTION`:** ввод не соответствует маске — строка **не сохранится** с таким паспортом.

---

### Задание 4 — красивый телефон

**Идея:** из введённого текста выкидываем всё, кроме цифр. Если 11 символов и начинается с `7` — считаем формат `7XXXXXXXXXX` и переписываем `NEW.phone` в вид `+7 (XXX) XXX XX XX`. Если 10 цифр — считаем, что нет ведущей семёрки, и подставляем `+7` сами.

**Почему `BEFORE`:** меняем значение **до** записи в таблицу — в базе уже лежит единый формат.

## 6. Что спросить могут на защите (коротко)

- **Чем `BEFORE` отличается от `AFTER`?** До записи — можно изменить данные или отменить; после — фиксируем факт и пишем в другие таблицы / лог.
- **Почему для запрета повторной продажи `EXCEPTION`, а для цен — `WARNING`?** В задании: повторную продажу **запретить**; по ценам — **предупредить**, но не обязательно рвать сделку.
- **Можно ли один триггер на всё?** Да, но в отчёте разнесено по заданиям — так проще читать и сдавать по пунктам.
- **В каком порядке срабатывают несколько `BEFORE` на одной таблице?** В PostgreSQL — по **алфавиту имён триггеров** (для одного момента времени и события). На защите достаточно знать, что порядок **фиксирован**, но **не стоит** на него опираться для бизнес-логики без явных имён/документации.
- **Зачем `EXECUTE PROCEDURE`?** В PostgreSQL триггер вызывает именно **функцию** с `RETURNS TRIGGER`; слово `PROCEDURE` здесь — часть синтаксиса `CREATE TRIGGER`, не путать с `CREATE PROCEDURE`.

---

## 7. Перед защитой — три фразы

1. **Триггер** — автоматический вызов функции при `INSERT`/`UPDATE`/`DELETE`; в функции доступны `NEW`/`OLD` и тип операции.
2. **Практика 3.1** — в основном проверки и предупреждения по продажам и комнатам плюс журнал изменений таблицы продаж.
3. **Практика 3.2** — бонусы риэлтора от суммы продаж с откатом при удалении, контроль потолка бонуса, проверка паспорта и нормализация телефона до вставки в таблицу.

Если нужно, можно отдельно расписать **построчно** только одно задание — напишите номер.
