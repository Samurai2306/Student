# Задание 1 — студенты со средним выше порога

## Таблицы и данные (построчно)

| Строка кода | Простыми словами |
|-------------|------------------|
| `DROP TABLE IF EXISTS grades;` | Если таблица оценок уже была — убрать её, чтобы скрипт можно было запускать заново без ошибки «таблица уже есть». Сначала `grades`, потому что она ссылается на студентов. |
| `DROP TABLE IF EXISTS students;` | То же для студентов. |
| `CREATE TABLE students(...)` | Таблица людей: номер и ФИО. |
| `id SERIAL PRIMARY KEY` | Номер сам растёт (1, 2, 3…), дубликатов id не бывает — это главный ключ. |
| `full_name TEXT` | ФИО как текст без жёсткой длины. |
| `CREATE TABLE grades(...)` | Таблица оценок: чья оценка, по какому предмету, какая цифра. |
| `student_id INTEGER REFERENCES students(id)` | Привязка к студенту: в базе нельзя воткнуть оценку на несуществующий id студента. |
| `subject TEXT`, `grade INTEGER` | Название предмета и балл. |
| `INSERT INTO students...` | Три тестовых студента — чтобы было на чём проверять запрос. |
| `INSERT INTO grades...` | По три оценки на каждого — чтобы средний считался осмысленно. |

## Функция `get_students_above_avg` (построчно)

| Строка / фрагмент | Простыми словами |
|-------------------|------------------|
| `CREATE OR REPLACE FUNCTION ...` | Создать функцию или, если с таким именем уже была — **заменить** тело на новое. |
| `(min_avg NUMERIC)` | Вход: одно число — **порог среднего** (например 4.0). |
| `RETURNS TABLE(student_id, full_name, avg_grade)` | На выходе не одно число, а **набор строк**, как у обычной таблицы: id, имя, средний балл. |
| `LANGUAGE sql` | Внутри функции — обычный SQL-запрос, без «если-то», переменных цикла и т.п. |
| `AS $$ ... $$` | Границы тела функции (удобно, когда внутри много строк). |
| `SELECT s.id AS student_id` | В результат пойдёт id студента под понятным именем столбца. |
| `s.full_name` | ФИО из таблицы студентов. |
| `AVG(g.grade)::NUMERIC AS avg_grade` | Среднее по всем оценкам этого студента; `::NUMERIC` — явно привести тип к «нормальному» числу с фиксированной логикой для сравнения с `min_avg`. |
| `FROM students s` | Берём студентов, короткий псевдоним `s`. |
| `JOIN grades g ON g.student_id = s.id` | К каждому студенту цепляем **только его** строки оценок. Если у кого-то нет оценок — он **вообще не появится** в этом соединении. |
| `GROUP BY s.id, s.full_name` | «Схлопываем» много строк оценок **в одну строку на студента», чтобы `AVG` имел смысл. `full_name` в GROUP BY — потому что мы его выводим и не оборачиваем в AVG. |
| `HAVING AVG(g.grade)::NUMERIC > min_avg` | **После** подсчёта среднего отсеиваем тех, кому среднее **не подошло**. Сюда нельзя перенести в `WHERE`: до группировки среднего ещё нет. |
| `ORDER BY avg_grade DESC, s.id` | Сначала с большим средним вверху; если средние равны — порядок по id, чтобы не прыгало. |

## Коротко: что спросят на защите

- **Почему не `WHERE AVG(...) > ...`?** Потому что `WHERE` работает по **отдельным строкам** до группировки, а среднее — это уже **итог по группе**. Для итога — только `HAVING`.
- **`>` или `>=`?** У тебя **строго больше**: ровно 4.0 при пороге 4.0 **не проходит**.
- **Студент без оценок?** В этом решении он **пропадёт** из-за обычного `JOIN`.

# Задание 2 — все подчинённые (рекурсия)

## Таблица и вставки (построчно)

| Строка / фрагмент | Простыми словами |
|-------------------|------------------|
| `DROP TABLE IF EXISTS employees;` | Чистый старт для перезапуска скрипта. |
| `id SERIAL PRIMARY KEY` | У каждого сотрудника свой уникальный номер. |
| `name VARCHAR(100) NOT NULL` | Имя, до 100 символов, пустым быть не может. |
| `manager_id INTEGER REFERENCES employees(id)` | Кто начальник: ссылка на **другую строку этой же таблицы**. У директора начальника нет — туда пишут `NULL`. |
| Комментарии `-- id = 1` и т.д. | Подсказка: какие номера получатся **если таблица пустая** и вставка идёт в таком порядке. Если в базе уже были люди — номера могут быть другие; для экзамена обычно гоняют с нуля. |

## Функция `get_subordinates_hierarchy` (построчно)

| Строка / фрагмент | Простыми словами |
|-------------------|------------------|
| `(p_employee_id INTEGER)` | С какого человека начинаем смотреть дерево (его id). |
| `RETURNS TABLE(employee_id, employee_name, level)` | Снова таблица из строк: кто попал, как зовут, на каком уровне. |
| `WITH RECURSIVE hierarchy AS (...)` | Заводим временный «результат по шагам» с именем `hierarchy`. **RECURSIVE** значит: вторую часть можно строить **от результата первой**, и так по кругу, пока есть кому добавлять. |
| **Первая часть (до `UNION ALL`)** | **Старт:** одна строка — тот самый сотрудник с id = `p_employee_id`, уровень **0**. Это «корень» для данного запуска. |
| `0 AS level` | Договорились: сам выбранный человек — нулевой этаж. |
| `WHERE e.id = p_employee_id` | Берём ровно одного. |
| `UNION ALL` | Склеиваем **старые** строки результата с **новыми**, которые мы сейчас дорисуем. |
| **Вторая часть (после `UNION ALL`)** | Ищем в `employees` всех, у кого `manager_id` равен **id кого-то из уже попавших в `hierarchy`**. Это и есть «подчинённые текущего набора». |
| `h.level + 1` | На каждом шаге вниз уровень на единицу больше. |
| `JOIN hierarchy h ON sub.manager_id = h.employee_id` | «Подчинённый» (`sub`) привязан к строке из уже накопленной иерархии как к своему начальнику. |
| Финальный `SELECT ... FROM hierarchy` | Выводим всё, что накопилось: сам человек + все ниже по цепочке. |
| `ORDER BY level, employee_id` | Сначала этаж 0, потом 1, потом 2… Внутри этажа — по id, чтобы порядок был стабильный. |

## Простая аналогия

Представь: на доске пишешь фамилию директора. Потом добавляешь всех, у кого в анкете «начальник — из тех, кто уже на доске». Потом снова — пока новых не останется. Уровень — это «сколько звеньев от первого имени вниз».

## Коротко: что спросят

- **Почему level с нуля?** Так в коде задумано: 0 — **сам сотрудник**, дальше вниз растёт. Если скажут «нам только подчинённые без него» — в конце добавляют `WHERE level > 0`.
- **Цикл в данных (А начальник Б, Б начальник А)?** Рекурсия может «сломаться» или упереться в лимит; в учебной задаче так не делают.

# Задание 3 — цикл, +10% к цене, число записей

## Таблица и вставки (построчно)

| Строка / фрагмент | Простыми словами |
|-------------------|------------------|
| `CREATE TABLE products(...)` | Товар: номер, название, цена. |
| `price NUMERIC` | Цена как «нормальное» десятичное число, не float с глюками. |
| Строка с `NULL` | Специально проверка: **нет цены** — умножать не на что / не нужно. |

## Функция `increase_all_product_prices_10pct` (построчно)

| Строка / фрагмент | Простыми словами |
|-------------------|------------------|
| `RETURNS INTEGER` | Наружу отдаём **одно целое число** — то, что накопили в счётчике. |
| `LANGUAGE plpgsql` | Нужен **процедурный** стиль: переменные, цикл, пошаговая логика. |
| `DECLARE` | Здесь объявляют переменные до основного кода. |
| `rec RECORD` | Одна переменная «текущая строка»: в неё по очереди попадут поля из выборки. |
| `processed_count INTEGER := 0` | Счётчик с нуля. |
| `BEGIN` … `END` | Тело выполнения. |
| `FOR rec IN SELECT id, price FROM products ORDER BY id` | Идём **по каждой строке** товаров в порядке id. На каждом круге в `rec` лежат `id` и `price` этой строки. |
| `LOOP` … `END LOOP` | Один проход тела на одну строку выборки. |
| `processed_count := processed_count + 1` | **Каждый раз**, как взяли новую строку, счётчик +1 — **даже если цена NULL** (см. ниже). |
| `UPDATE products SET price = ROUND(price * 1.10, 2)` | Цена стала **старая × 1.1**, округлили до **2 знаков после запятой** (как копейки). |
| `WHERE id = rec.id AND price IS NOT NULL` | Меняем **только эту** строку по id; и только если цена не пустая — иначе `UPDATE` по сути ничего не изменит (условие не выполнится). |
| `RETURN processed_count` | Вернули, сколько раз крутился цикл по строкам. |

## Важный нюанс (честно сказать ревьюеру)

Счётчик растёт **на каждую строку в цикле**. Строка без цены всё равно **увеличит счётчик**, но **цена у неё не обновится** из-за `price IS NOT NULL`. Если спросят «почему число не совпало с числом обновлённых» — вот причина; «исправить» можно, если считать только успешные обновления — другим кодом.

## Коротко: что спросят

- **Зачем цикл, если можно одним UPDATE?** В задании просили **перебор**; один UPDATE — другое решение, его упоминают как альтернативу.
- **Повторный вызов?** Цены снова ×1.1 — это **не** «поднять ровно один раз навсегда», а «каждый раз пересчитать от текущего».

## Шпаргалка терминов (если попросят одним словом)

- **PRIMARY KEY** — главный id строки, не повторяется.
- **REFERENCES** — внешний ключ: нельзя сослаться на несуществующую строку.
- **SERIAL** — id сам увеличивается при вставке.
- **JOIN** — склеить таблицы по условию.
- **GROUP BY + AVG** — сжать много оценок в одно среднее на человека.
- **HAVING** — отфильтровать **уже посчитанные** группы (например по среднему).
- **WITH RECURSIVE** — иерархия «шаг за шагом» вниз или вверх.
- **plpgsql** — язык функций с циклами и переменными.
