## Задание 1 — Вставка, обновление и удаление строки (MVCC, `xmin/xmax`)

### Шаг 1 — вставка первой строки (Терминал A)

```sql
INSERT INTO a(id) VALUES (1);
SELECT xmin, xmax, * FROM a;
```

**Что увидеть:** `xmin` — идентификатор транзакции, создавшей видимую версию строки. `xmax` обычно `0` (строка не помечена на удаление/обновление следующей версией).

### Шаг 2 — явная транзакция и обновление (Терминал A)

```sql
BEGIN;
UPDATE a SET id = 2 WHERE id = 1;
SELECT xmin, xmax, * FROM a;
```

**Идея:** `UPDATE` в PostgreSQL — это «создать новую версию строки» + «пометить старую как невидимую для будущих снимков».

### Шаг 3 — наблюдение во второй сессии (Терминал B)

Пока транзакция A не зафиксирована, во второй сессии:

```sql
SELECT xmin, xmax, * FROM a;
```

**Что важно заметить:**

- В B вы, как правило, увидите **старое значение** `id` (т.к. новая версия ещё не зафиксирована).
- При этом `xmax` у старой версии может быть уже заполнен — это признак, что создана новая версия (но она ещё не стала видимой вашему снимку).

### Шаг 4 — фиксация (Терминал A) и повторное чтение (Терминал B)

Терминал A:

```sql
COMMIT;
```

Терминал B:

```sql
SELECT xmin, xmax, * FROM a;
```

Теперь B увидит **новую версию** строки.

---

### Шаг 5 — удаление как «версионная операция» (Терминал A и B)

Терминал A:

```sql
BEGIN;
DELETE FROM a WHERE id = 2;
SELECT xmin, xmax, * FROM a;
```

**Что увидеть в A:** строка «исчезла» из результата, потому что внутри своей транзакции вы видите собственные изменения.

Терминал B (пока A не зафиксировал):

```sql
SELECT xmin, xmax, * FROM a;
```

**Что увидеть в B:** строка ещё видна, но `xmax` может измениться (строка помечена на удаление транзакцией A, но для B удаление ещё не «наступило»).

Терминал A:

```sql
COMMIT;
```

Терминал B:

```sql
SELECT xmin, xmax, * FROM a;
```

Теперь строка исчезает и в B.

## Задание 2 — Видимость версий на разных уровнях изоляции

### Шаг 1 — проверить уровень изоляции (Терминал A)

```sql
BEGIN;
SHOW transaction_isolation;
```

Обычно по умолчанию будет `read committed`.

### Шаг 2 — вставить новую строку (Терминал A)

```sql
INSERT INTO a(id) VALUES (10);
```

### Шаг 3 — во второй транзакции не видно незакоммиченного (Терминал B)

```sql
BEGIN;
SHOW transaction_isolation;
SELECT * FROM a ORDER BY id;
```

Пока A не зафиксировал, в B новой строки не будет.

### Шаг 4 — фиксируем A, повторяем чтение в B (аномалия неповторяющегося чтения)

Терминал A:

```sql
COMMIT;
```

Терминал B (внутри той же транзакции уровня `read committed`):

```sql
SELECT * FROM a ORDER BY id;
COMMIT;
```

**Ожидаемый смысл:** во второй транзакции при повторном запросе вы увидите изменения, которые «пришли» после первого чтения — это демонстрация **неповторяющегося чтения** на `read committed` (снимок обновляется на каждый оператор).

---

### Шаг 5 — повторяем на `REPEATABLE READ` (один снимок на всю транзакцию)

Терминал A:

```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
INSERT INTO a(id) VALUES (20);
```

Терминал B:

```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT * FROM a ORDER BY id;
```

Терминал A:

```sql
COMMIT;
```

Терминал B (всё ещё в той же транзакции):

```sql
SELECT * FROM a ORDER BY id;
COMMIT;
```

**Что увидеть:** во второй транзакции B строка, вставленная A, **не появится** до `COMMIT` в B, т.к. `REPEATABLE READ` держит **один и тот же снимок** данных на всю транзакцию.

## Задание 3 — Состояние транзакции по CLOG (`pg_xact_status`)

> В методичке предлагается посмотреть статус конкретного `xid` (например, `1598`). У вас значения будут другими.

### Шаг 1 — получить номер текущей транзакции

Удобно зафиксировать свой `xid` прямо из сессии:

```sql
BEGIN;
SELECT txid_current() AS xid;
INSERT INTO a(id) VALUES (30);
```

Вместо `<xid_из_вывода>` подставьте число из `txid_current()`.

### Шаг 2 — статус «в процессе», затем «зафиксирована»

Пока транзакция не зафиксирована:

```sql
SELECT pg_xact_status('<xid_из_вывода>');
```

Затем:

```sql
COMMIT;
SELECT pg_xact_status('<xid_из_вывода>');
```

**Смысл:** увидеть переход статуса транзакции в CLOG (commit log).

### Шаг 3 — аналогично для отката

```sql
BEGIN;
SELECT txid_current() AS xid_rollback;
INSERT INTO a(id) VALUES (40);
SELECT pg_xact_status('<xid_rollback>');
ROLLBACK;
SELECT pg_xact_status('<xid_rollback>');
```

**Смысл:** после `ROLLBACK` статус будет отражать «откачена/aborted».

## Задание 4 — Блокировки таблицы (`pg_locks`)

Цель: увидеть, какие режимы блокировок ставятся на таблицу при `UPDATE`, и как они конфликтуют с операциями вроде `CREATE INDEX`.

### Шаг 1 — транзакция с обновлением и получение `pid` (Терминал A)

```sql
BEGIN;
SELECT pg_backend_pid() AS pid_a;
UPDATE a SET id = id + 1;
```

### Шаг 2 — посмотреть блокировки процесса A (можно в A, можно в отдельной сессии)

Подставьте свой `pid_a`:

```sql
SELECT locktype,
       transactionid,
       mode,
       relation::regclass AS obj
FROM pg_locks
WHERE pid = <pid_a>;
```

**Ожидаемая идея:** появится `RowExclusiveLock` на таблицу (типично для операций изменения данных).

### Шаг 3 — во второй сессии попытка создать индекс (Терминал B)

Терминал B:

```sql
BEGIN;
SELECT pg_backend_pid() AS pid_b;
CREATE INDEX ON a (id);
```

Команда `CREATE INDEX` может «подвиснуть», ожидая освобождения несовместимой блокировки.

### Шаг 4 — посмотреть блокировки процесса B и увидеть ожидание

Подставьте `pid_b`:

```sql
SELECT locktype,
       transactionid,
       mode,
       relation::regclass AS obj
FROM pg_locks
WHERE pid = <pid_b>;
```

**Смысл:** в B вы увидите блокировку в режиме вроде `ShareLock` (для DDL) и/или состояние ожидания из-за конфликта с блокировкой A.

### Шаг 5 — снять блокировку фиксацией A

Терминал A:

```sql
COMMIT;
```

Сразу после этого в Терминале B команда `CREATE INDEX` должна продолжиться и завершиться.

Терминал B:

```sql
COMMIT;
```

## Итог

В практике показано, что в PostgreSQL:

- `UPDATE/DELETE` работают через **версии строк** (MVCC), что наблюдается через `xmin/xmax`.
- На `READ COMMITTED` каждый оператор получает **новый снимок**, а на `REPEATABLE READ` транзакция работает с **одним снимком**.
- Статус транзакции можно посмотреть через `pg_xact_status(xid)` (CLOG).
- Блокировки (`pg_locks`) позволяют диагностировать «подвисания» и конфликтующие DDL/DML операции.
