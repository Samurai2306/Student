# Система мониторинга HTTP-запросов

Подробная документация к проекту `HttpRequestMonitor`.

## 1. Назначение проекта

`HttpRequestMonitor` - это WPF-приложение на .NET, которое совмещает в одном интерфейсе:

- локальный HTTP-сервер на базе `HttpListener`;
- HTTP-клиент на базе `HttpClient`;
- мониторинг входящих и исходящих запросов;
- логирование запросов в интерфейс и файл;
- расчет базовой статистики;
- визуализацию нагрузки по времени через график.

Проект предназначен для демонстрации работы с сетевыми запросами, асинхронным программированием, многопоточной обработкой входящих подключений, WPF-интерфейсом и простым анализом HTTP-трафика.

## 5. Основные возможности

Приложение реализует следующие функции:

- запуск и остановку локального HTTP-сервера;
- настройку порта сервера из интерфейса;
- обработку `GET`-запросов;
- обработку `POST`-запросов с JSON-телом;
- сохранение полученных сообщений в памяти приложения;
- отправку клиентских `GET`-запросов на локальные или внешние API;
- отправку клиентских `POST`-запросов с JSON-телом;
- отображение ответа сервера в интерфейсе;
- логирование серверных и клиентских событий;
- фильтрацию логов по HTTP-методу;
- фильтрацию логов по группе HTTP-статусов;
- подсчет общего количества `GET` и `POST` запросов;
- расчет среднего времени обработки запросов;
- построение графика нагрузки по минутам;
- сохранение логов в файл `logs.txt`.

## 6. Архитектура приложения

Проект построен в стиле MVVM.

Основные слои:

- `Models` - модели данных.
- `Services` - инфраструктурная логика HTTP-сервера.
- `ViewModels` - состояние интерфейса, команды, клиентские запросы, статистика и логи.
- `Views` - WPF-разметка главного окна.

### 6.1. Models

Папка:

```text
HttpRequestMonitor/Models
```

Содержит простые классы данных.

#### `Message.cs`

Модель сообщения, полученного сервером через `POST`.

Поля:

- `Id` - уникальный идентификатор сообщения типа `Guid`;
- `Text` - текст сообщения;
- `ReceivedAt` - время получения сообщения.

Сообщения создаются сервером после успешного разбора JSON-тела вида:

```json
{
  "message": "Hello from WPF"
}
```

#### `LogEntry.cs`

Модель записи журнала.

Поля:

- `Timestamp` - время события;
- `Source` - источник события, например `Server` или `Client`;
- `Method` - HTTP-метод;
- `Url` - URL запроса;
- `StatusCode` - HTTP-код ответа;
- `ProcessingTimeMs` - время обработки в миллисекундах;
- `Headers` - заголовки запроса;
- `RequestBody` - тело запроса;
- `ResponseBody` - тело ответа;
- `DisplayText` - готовое текстовое представление для вывода в журнал.

Эта модель используется одновременно для таблицы логов, текстового журнала, статистики и записи в файл.

### 6.2. Services

Папка:

```text
HttpRequestMonitor/Services
```

#### `HttpServer.cs`

Класс инкапсулирует всю серверную логику.

Основные обязанности:

- создание и настройка `HttpListener`;
- запуск прослушивания адресов;
- остановка сервера;
- прием входящих запросов;
- многопоточная обработка запросов;
- обработка `GET`;
- обработка `POST`;
- валидация JSON;
- хранение принятых сообщений;
- публикация логов через событие `RequestLogged`;
- расчет серверной статистики для `GET`-ответа.

Сервер слушает два префикса:

```text
http://localhost:{port}/
http://127.0.0.1:{port}/
```

Например, при порте `8080`:

```text
http://localhost:8080/
http://127.0.0.1:8080/
```

Сервер запускается в фоновом режиме и не блокирует UI.

## 7. HTTP-сервер

### 7.1. Запуск сервера

В интерфейсе есть блок `Управление сервером`.

Порядок запуска:

1. Ввести порт, например `8080`.
2. Нажать кнопку `Запустить`.
3. Убедиться, что статус изменился на:

```text
Сервер запущен: http://localhost:8080/
```

Если порт введен некорректно, приложение покажет сообщение:

```text
Введите корректный порт от 1 до 65535.
```

### 7.2. Остановка сервера

Для остановки нужно нажать кнопку `Остановить`.

После остановки статус меняется на:

```text
Сервер остановлен
```

### 7.3. GET-запрос

`GET`-запрос возвращает информацию о состоянии сервера.

Пример URL:

```text
http://localhost:8080/
```

Пример ответа:

```json
{
  "status": "running",
  "uptime": "00:01:23.4567890",
  "totalRequests": 5,
  "getRequests": 3,
  "postRequests": 2,
  "averageProcessingTimeMs": 2.4,
  "storedMessages": 2
}
```

Поля ответа:

- `status` - состояние сервера;
- `uptime` - время работы сервера с момента запуска;
- `totalRequests` - общее количество обработанных сервером запросов;
- `getRequests` - количество `GET`-запросов;
- `postRequests` - количество `POST`-запросов;
- `averageProcessingTimeMs` - среднее время обработки серверных запросов;
- `storedMessages` - количество сообщений, сохраненных из `POST`-запросов.

### 7.4. POST-запрос

`POST`-запрос принимает JSON с обязательным строковым полем `message`.

Пример тела:

```json
{
  "message": "Привет от клиента"
}
```

При успешной обработке сервер:

1. Разбирает JSON через `System.Text.Json`.
2. Проверяет наличие непустого поля `message`.
3. Создает объект `Message`.
4. Присваивает ему уникальный `Guid`.
5. Сохраняет сообщение в памяти.
6. Возвращает клиенту ID сообщения и время получения.

Пример успешного ответа:

```json
{
  "id": "f0bbbf97-405f-4f31-b3b7-16fcbb83f598",
  "receivedAt": "2026-04-27T10:00:00.0000000+03:00"
}
```

Успешный `POST` возвращает HTTP-статус:

```text
201 Created
```

### 7.5. Ошибки POST-запроса

Если JSON некорректный:

```json
{
  "error": "Request body must be valid JSON."
}
```

Если поле `message` отсутствует или пустое:

```json
{
  "error": "JSON body must contain a non-empty string property 'message'."
}
```

В обоих случаях сервер возвращает:

```text
400 Bad Request
```

### 7.6. Неподдерживаемые методы

Если отправить метод, отличный от `GET` или `POST`, сервер вернет:

```json
{
  "error": "Supported methods: GET, POST."
}
```

HTTP-статус:

```text
405 Method Not Allowed
```

## 8. HTTP-клиент

Клиентская часть находится в правом верхнем блоке интерфейса `HTTP-клиент`.

Элементы:

- поле ввода URL;
- выпадающий список метода (`GET` или `POST`);
- редактор JSON-тела;
- кнопка `Отправить`;
- поле вывода ответа.

### 8.1. GET-запрос через интерфейс

Пример:

1. Запустить сервер на порту `8080`.
2. В поле URL указать:

```text
http://localhost:8080/
```

3. В списке методов выбрать `GET`.
4. Нажать `Отправить`.

В поле ответа появится JSON со статусом сервера.

Также можно отправлять запросы на внешние публичные API, например:

```text
https://api.github.com/
```

Важно: некоторые публичные API требуют дополнительные заголовки, авторизацию или конкретные пути. Если API отказывает в запросе, это не ошибка приложения.

### 8.2. POST-запрос на локальный сервер

Пример:

1. Запустить сервер на порту `8080`.
2. В поле URL указать:

```text
http://localhost:8080/
```

3. Выбрать метод `POST`.
4. В JSON-редакторе указать:

```json
{
  "message": "Hello from WPF"
}
```

5. Нажать `Отправить`.

В ответе должен появиться JSON с `id` и `receivedAt`.

### 8.3. Валидация URL

Перед отправкой клиент проверяет, что URL:

- является абсолютным;
- использует схему `http` или `https`.

Если URL некорректный, в поле ответа будет:

```text
Введите корректный HTTP/HTTPS URL.
```

### 8.4. Валидация JSON

Для `POST` клиент предварительно проверяет, что тело является корректным JSON.

Если JSON некорректный, в поле ответа будет:

```text
JSON-тело POST-запроса некорректно.
```

## 9. Мониторинг и аналитика

Приложение собирает информацию обо всех событиях, которые проходят через серверную и клиентскую части.

### 9.1. Источники логов

В логах есть поле `Source`.

Возможные значения:

- `Server` - событие пришло от встроенного HTTP-сервера;
- `Client` - событие пришло от встроенного HTTP-клиента.

Если клиент отправляет запрос на локальный сервер приложения, в логах обычно появятся две записи:

- одна клиентская запись `Client`;
- одна серверная запись `Server`.

Это ожидаемое поведение, потому что приложение фиксирует обе стороны обмена.

### 9.2. Счетчики GET и POST

В блоке `Статистика` отображаются:

- количество `GET`-запросов;
- количество `POST`-запросов;
- среднее время обработки.

Статистика считается по коллекции логов приложения.

Формула среднего времени:

```text
Tavg = сумма времени обработки всех записей / количество записей
```

### 9.3. Среднее время обработки

Время обработки считается в миллисекундах.

Для серверного события это время от начала обработки входящего запроса до завершения подготовки ответа.

Для клиентского события это время от начала отправки запроса до получения ответа или возникновения ошибки.

### 9.4. График нагрузки

График `Пиковая нагрузка по минутам` строится через `OxyPlot.Wpf`.

Логика построения:

1. Все логи группируются по минутам.
2. Для каждой минуты считается количество событий.
3. На графике отображается линия количества запросов во времени.

Ось X:

```text
Время, формат HH:mm
```

Ось Y:

```text
Количество запросов
```

## 10. Логирование

Логирование выполняется в двух местах:

- в интерфейсе приложения;
- в файле `logs.txt`.

### 10.1. Таблица логов

В таблице отображаются:

- время;
- источник;
- метод;
- статус;
- время обработки в миллисекундах;
- URL.

Таблица предназначена для быстрого просмотра событий.

### 10.2. Текстовый журнал

Текстовый журнал показывает более компактное текстовое представление события.

Пример:

```text
[2026-04-27 10:00:00] Server POST http://localhost:8080/ -> 201 (3.5 ms)
Body: {
  "message": "Hello from WPF"
}
```

### 10.3. Фильтрация логов

Доступны два фильтра:

- по методу;
- по статусу ответа.

Фильтр по методу:

- `Все`
- `GET`
- `POST`

Фильтр по статусу:

- `Все`
- `2xx`
- `3xx`
- `4xx`
- `5xx`

Фильтры применяются к таблице логов. Текстовый журнал остается полной хронологической лентой событий.

### 10.4. Файл logs.txt

Файл логов создается в выходной директории приложения:

```text
P_3/HttpRequestMonitor/bin/Debug/net8.0-windows/logs.txt
```

При запуске через другую конфигурацию путь будет соответствовать этой конфигурации, например:

```text
bin/Release/net8.0-windows/logs.txt
```

Логи записываются:

- автоматически при каждом событии;
- вручную по кнопке `Сохранить`;
- при закрытии приложения.

Формат строки в файле:

```text
Timestamp | Source | Method | Url | StatusCode | ProcessingTime | Headers | RequestBody | ResponseBody
```

Пример:

```text
2026-04-27T10:00:00.0000000+03:00 | Server | POST | http://localhost:8080/ | 201 | 3.5 ms | Headers: Content-Type: application/json | RequestBody: {"message":"Hello"} | ResponseBody: {"id":"...","receivedAt":"..."}
```

## 11. Асинхронность и потоки

Приложение использует асинхронную модель выполнения.

### 11.1. Сервер

`HttpServer` запускает прослушивание в фоновой задаче.

Каждый входящий запрос передается на отдельную обработку:

```text
Listener loop -> GetContextAsync -> ProcessRequestAsync
```

Это позволяет серверу принимать несколько запросов одновременно.

### 11.2. UI-поток

События сервера приходят из фоновых потоков, поэтому обновление WPF-коллекций выполняется через:

```csharp
Application.Current.Dispatcher.InvokeAsync(...)
```

Это важно, потому что `ObservableCollection` и WPF-controls нельзя безопасно обновлять напрямую из фонового потока.

### 11.3. Запись в файл

Запись в `logs.txt` защищена через `SemaphoreSlim`.

Это предотвращает одновременную запись нескольких событий в файл и уменьшает риск повреждения строк лога при параллельных запросах.

- `ProcessPostBody(...)` - обрабатывает тело `POST`;
- `PublishLogAsync(...)` - отправляет событие лога во ViewModel;
- `ResetState()` - сбрасывает серверную статистику при новом запуске.

## 13. Типовые сценарии проверки

### 13.1. Проверка запуска сервера

1. Запустить приложение.
2. Указать порт `8080`.
3. Нажать `Запустить`.
4. Проверить статус сервера.
5. Отправить `GET` на `http://localhost:8080/`.

Ожидаемый результат:

- в ответе JSON со статусом сервера;
- в логах появилась запись `Client GET`;
- в логах появилась запись `Server GET`, если запрос отправлен встроенным клиентом на локальный сервер.

### 13.2. Проверка POST

1. Запустить сервер.
2. В клиенте указать `http://localhost:8080/`.
3. Выбрать `POST`.
4. Ввести:

```json
{
  "message": "Test message"
}
```

5. Нажать `Отправить`.

Ожидаемый результат:

- HTTP-статус `201`;
- в ответе есть `id`;
- в ответе есть `receivedAt`;
- счетчик `POST` увеличился;
- в таблице логов появились записи.

Ожидаемый результат:

## 14. Особенности поведения

### 14.1. Почему один запрос может дать две записи в логах

Если встроенный клиент отправляет запрос на встроенный сервер, приложение фиксирует:

- факт отправки запроса клиентом;
- факт обработки запроса сервером.

Поэтому в логах появляются две записи.

Это нормально и полезно для мониторинга полного цикла запроса.

### 14.2. Почему статус может быть `0`

Для клиентского запроса статус `0` означает, что HTTP-ответ не был получен.

Например:

- сервер недоступен;
- URL неверный;
- сеть недоступна;
- соединение было прервано;
- внешний API не ответил.

В этом случае текст ошибки выводится в поле ответа и сохраняется в `ResponseBody` лога.

### 14.3. Почему статус может быть `499`

В серверной части статус `499` используется как технический статус для ситуации, когда клиент закрыл соединение до завершения отправки ответа.

Это не стандартный статус .NET, но он часто используется как обозначение:

```text
Client Closed Request
```

### 14.4. Где хранятся сообщения POST

Сообщения, полученные через `POST`, хранятся в памяти процесса в `ConcurrentDictionary`.

Они не сохраняются между запусками приложения.

В файл сохраняются только логи.

## 15. Возможные проблемы и решения

### 15.1. Сервер не запускается

Возможные причины:

- порт уже занят другим процессом;
- введен некорректный порт;
- приложение не имеет прав открыть указанный HTTP-префикс;
- предыдущий экземпляр приложения еще запущен.

Что попробовать:

1. Использовать другой порт, например `8081`.
2. Закрыть другие экземпляры приложения.
3. Проверить, не занят ли порт.

Команда PowerShell:

```powershell
netstat -ano | findstr :8080
```

### 15.2. GET на localhost не отвечает

Проверьте:

- сервер запущен;
- порт в URL совпадает с портом сервера;
- URL начинается с `http://`, а не `https://`;
- в URL указан правильный адрес:

```text
http://localhost:8080/
```

### 15.3. POST возвращает 400

Проверьте, что JSON имеет именно поле `message`:

```json
{
  "message": "Текст сообщения"
}
```

Поле должно быть строкой и не должно быть пустым.

### 15.4. Внешний API не отвечает

Возможные причины:

- API требует авторизацию;
- API требует специальные заголовки;
- API блокирует запросы без `User-Agent`;
- нет доступа к интернету;
- URL указан не полностью.

Для проверки лучше использовать простые публичные endpoints, которые не требуют авторизации.

### 15.5. Файл logs.txt не найден

Файл создается после первого события или после нажатия `Сохранить`.

Проверьте папку, соответствующую текущей конфигурации сборки:

```text
bin/Debug/net8.0-windows/
```

или:

```text
bin/Release/net8.0-windows/
```

`HttpRequestMonitor` реализует учебную систему мониторинга HTTP-запросов, объединяющую сервер, клиент, логирование, аналитику и визуализацию в одном WPF-приложении.

Проект демонстрирует:

- работу с HTTP на стороне сервера и клиента;
- асинхронное программирование;
- многопоточную обработку запросов;
- безопасное обновление WPF-интерфейса;
- сериализацию и валидацию JSON;
- построение графиков;
- базовую архитектуру MVVM;
- сохранение диагностической информации в файл.
