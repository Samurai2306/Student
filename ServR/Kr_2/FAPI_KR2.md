## 🌟 Задание 5.3

Это задание **требует доработки вашего предыдущего решения** с аутентификацией на основе куки. Вам предстоит реализовать безопасный механизм сессий, который автоматически продлевает срок их действия при активности пользователя и гарантирует защиту от подделки данных. Для успешного выполнения потребуется самостоятельное изучение документации и анализ возможностей библиотек.

### Требования к доработке:

1. **Динамическое время жизни сессии**

    - Сессия автоматически завершается через **5 минут** после **последней активности пользователя.**
    - При **обращении** к защищённому маршруту время сессии **сбрасывается** и продлевается **на 5 минут** с момента запроса, если прошло **более (и равно) 3 и менее 5 минут** с момента последней активности.
    - Пример:
        - Логин в 10:00 → сессия действует до 10:05.
        - Запрос в 10:02 → сессия не продлевается (прошло менее 3 минут с последней активности).
        - Запрос в 10:03 → сессия продлевается до 10:08 (новые 5 минут).
        - Запрос в 10:05 → сессия не продлевается (менее 3 минут с активности в 10:03).
        - Запрос в 10:07 → сессия продлевается до 10:12.
        - Если следующий запрос в 10:16 → доступ запрещён (прошло 9 минут с последней активности).

2. **Формат куки `session_token`**
    
    - Должен содержать:
        - Уникальный идентификатор пользователя (например, `UUID`) — оставляем предыдущую реализацию.
        - Время последней активности (UNIX timestamp).
        - Криптографическую подпись, защищающую **оба значения** от изменений.
    - Пример значения: `d92d5e6e-98a1-4c2a-b3c7-5d5e8d7f8e9a.1715000400.signature_abc123`.

    ```html
    <user_id>.<timestamp>.<signature>
    ```

3. **Проверка целостности данных**

    - Любая попытка изменить идентификатор или время в куке должна немедленно блокироваться.
    - Сервер обязан проверять:
        - Корректность подписи.
        - Соответствие времени в куке серверному времени.

4. **Обновление куки**

    - При каждом успешном обращении к защищённому маршруту:
        - Если с момента последней активности прошло **менее 3 минут** → кука **НЕ** обновляется с новым временем.
        - Если с момента последней активности прошло **менее 5 минут, но более 3 минут** → кука обновляется с новым временем.
        - Если прошло **более 5 минут** → сессия завершается (статус 401).

### Что нужно реализовать:

1. **Маршрут** `/login`
    - Устанавливает куку `session_token` с текущим временем активности.
    - Параметры куки: `httponly=True`, `secure=False` (для тестирования, в продакшене `True`), `max_age=300` (5 минут).

2. **Защищённые маршруты** (например, `/profile`)
    - Проверяют целостность данных и актуальность сессии.
    - При успешной проверке:
        - Возвращают информацию профиля.
        - Обновляют куку, если это необходимо.

3. **Обработка ошибок**

    - Недействительная/просроченная сессия → статус 401 с `{"message": "Session expired"}`.
    - Поддельные данные → статус 401 с `{"message": "Invalid session"}`.

**Критерии успешности:**

1. Сессия не продлевается, если с момента последней активности прошло больше 5 минут.
2. Время в куке синхронизировано с серверным временем.
3. Любое изменение данных в куке (идентификатор, время, подпись) → немедленный запрет доступа.

**Рекомендации для тестирования:**

- Используйте **Postman** с функцией задержки между запросами (например, 2 минуты).
- Попробуйте вручную изменить значение `session_token` в браузере, чтобы проверить реакцию сервера.
- Проверьте сценарии:
    - Логин → запрос через 2 минуты → кука не обновляется.
    - Логин → запрос через 4 минуты → кука обновляется.
    - Логин → запрос через 6 минут → ответ со статус кодом 401.
    - Изменение идентификатора в куке → ответ со статус кодом 401.

Чтобы изменить статус-код ответа, можно воспользоваться объектом класса `Response` из **FastAPI** (работу с ошибками мы пройдем позднее):

```python
@app.get("/example")
def example(response: Response):
    response.status_code = 401
    return {"message": "Unauthorized"}
```

**2. Реализуйте два маршрута с использованием модели CommonHeaders:**

- **Маршрут `/headers` (метод GET):**
    - Внедрите модель **CommonHeaders** в маршрут, чтобы она принимала заголовки из модели.
    - Верните JSON-ответ, содержащий извлечённые значения заголовков **User-Agent** и **Accept-Language**.

- **Маршрут `/info` (метод GET):**
    - Аналогично, подключите модель **CommonHeaders** для извлечения заголовков.
    - Верните заголовки в теле ответа (в формате JSON), а также расширьте ответ полем `message` с текстом: "`Добро пожаловать! Ваши заголовки успешно обработаны.`"
    - Помимо возврата заголовков в теле ответа, добавьте в ответ HTTP-заголовок **X-Server-Time**, в котором будет возвращаться текущее серверное время.

**3. Обработка ошибок:**

[Опционально] Если заголовок **Accept-Language** имеет неверный формат, оба маршрута должны возвращать ответ с описанием ошибки.

**Примеры запросов и ответов:**

Запрос к `/headers`:

**Запрос:**

```bash
GET /headers HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept-Language: en-US,en;q=0.9,es;q=0.8
```

**Ответ:**

```json
{
    "User-Agent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64)",
    "Accept-Language": "en-US,en;q=0.9,es;q=0.8"
}
```

Запрос к `/info`:

**Запрос:**

```bash
GET /info HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept-Language: en-US,en;q=0.9,es;q=0.8
```

**Ответ:**

- Тело ответа (JSON):

```json
{
    "message": "Добро пожаловать! Ваши заголовки успешно обработаны.",
    "headers": {
        "User-Agent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64)",
        "Accept-Language": "en-US,en;q=0.9,es;q=0.8"
    }
}
```

- Дополнительно, в заголовках ответа должен присутствовать ключ **X-Server-Time** с текущей датой и временем, например:

```bash
X-Server-Time: 2025-04-16T12:34:56
```
