# Practice 11 — как всё работает

Этот файл объясняет логику практики 11 простым языком: что делает бэкенд, как работает RBAC, как устроен фронтенд и какой сценарий работы пользователя.

## 1) Что реализовано по заданию

Практика 11 = доработка практики 10:

- аутентификация (`register`, `login`, `refresh`, `me`);
- авторизация по ролям (RBAC);
- управление товарами;
- управление пользователями (для администратора).

Роли в системе:

- `user` — просмотр товаров;
- `seller` — права `user` + создание/редактирование товаров;
- `admin` — права `seller` + удаление товаров + управление пользователями.

## 2) Структура проекта

- `server/app.js` — весь сервер (Express, JWT, RBAC, маршруты);
- `client/src/api/*` — HTTP-клиент и методы API;
- `client/src/state/AuthContext.jsx` — глобальное состояние авторизации;
- `client/src/pages/*` — страницы интерфейса;
- `client/src/components/Layout.jsx` + `layout.css` — оболочка приложения и общие стили;
- `client/src/index.css` — цветовая тема (сейчас лавандовая).

## 3) Бэкенд: основные идеи

### 3.1 Хранилище данных

Используется in-memory хранилище (массивы в памяти процесса):

- `users` — пользователи;
- `products` — товары;
- `refreshTokens` — активные refresh-токены.

Важно: данные живут только пока работает сервер. После перезапуска очищаются.

### 3.2 Токены

- Access token (короткий) — для доступа к защищенным маршрутам.
- Refresh token (длинный) — для обновления пары токенов.

Refresh-токен хранится на сервере в `Set`, а на клиенте в `localStorage`.

### 3.3 Middleware

- `authMiddleware`:
  - читает `Authorization: Bearer <token>`;
  - валидирует access token;
  - проверяет, что пользователь существует и не заблокирован;
  - кладет пользователя в `req.user`.

- `roleMiddleware([...roles])`:
  - пропускает запрос только если роль пользователя входит в список.

### 3.4 Валидация и защита

Добавлена базовая «гигиена»:

- валидация `email`;
- минимальная длина пароля;
- проверки непустых строк;
- невозможность самоблокировки админа;
- при блокировке пользователя или смене пароля — отзыв его refresh-токенов;
- регистрация гостя создает только роль `user` (без эскалации в `admin`).

## 4) Маршруты и права доступа

### 4.1 Auth

- `POST /api/auth/register` — гость;
- `POST /api/auth/login` — гость;
- `POST /api/auth/refresh` — гость (по refresh токену в заголовке);
- `GET /api/auth/me` — авторизованный пользователь (`user|seller|admin`).

### 4.2 Users (только admin)

- `GET /api/users` — список пользователей;
- `GET /api/users/:id` — пользователь по id;
- `PUT /api/users/:id` — обновление полей пользователя;
- `DELETE /api/users/:id` — блокировка пользователя.

### 4.3 Products

- `POST /api/products` — `seller|admin`;
- `GET /api/products` — `user|seller|admin`;
- `GET /api/products/:id` — `user|seller|admin`;
- `PUT /api/products/:id` — `seller|admin`;
- `DELETE /api/products/:id` — `admin`.

## 5) Фронтенд: как устроен поток

### 5.1 AuthContext

`AuthContext` хранит текущего пользователя и методы:

- `login`;
- `register`;
- `logout`;
- `refreshUser`.

При старте приложения делается `me()`, чтобы восстановить сессию, если токены уже есть.

### 5.2 API-клиент

В `client/src/api/client.js`:

- request-interceptor автоматически подставляет access token;
- response-interceptor при `401` пытается сделать `refresh`;
- если refresh успешен — повторяет исходный запрос;
- если нет — очищает токены.

### 5.3 Страницы

- `Products` — список товаров;
- `ProductCreate` — создание (только seller/admin);
- `ProductDetail` — просмотр;
- `ProductEdit` — редактирование и удаление (удаление только admin);
- `Users` — админская страница управления пользователями;
- `Login` / `Register` — вход и регистрация.

После успешного создания товара пользователь переводится на `/products` (список всех товаров).

## 6) Тестовый администратор

При запуске сервера создается demo-admin:

- `email`: `admin@practice11.local`
- `password`: `admin123`

Можно переопределить через env:

- `ADMIN_EMAIL`
- `ADMIN_PASSWORD`

## 7) Как запускать

Сервер:

```bash
cd Practice_11/server
npm i
npm run start
```

Клиент:

```bash
cd Practice_11/client
npm i
npm run dev
```
